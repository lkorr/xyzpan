## Summary

The existing `getStateInformation`/`setStateInformation` implementation in `PluginProcessor.cpp` already uses the correct JUCE APVTS pattern: `apvts.copyState()` in save, `apvts.replaceState(juce::ValueTree::fromXml(*xml))` in restore. This is the standard pattern recommended by JUCE docs and the community. The pattern is thread-safe (uses locks internally) but NOT realtime-safe — which is fine because these methods are never called from the audio thread.

The previous segfault was almost certainly caused by a prior implementation that called memory-allocating code from within `processBlock()` or from a path that could be concurrent with the audio thread without protection. The APVTS atomic pointer pattern (`getRawParameterValue()` returning `std::atomic<float>*`) is the only safe way to read parameters on the audio thread, and the existing code already uses this correctly. The segfault path was likely: some code that called `replaceState()` or allocated heap memory while the audio thread was running, causing lock inversion or heap corruption.

For factory presets, the correct approach is to embed preset definitions as XML strings in a `Presets.h` file (no BinaryData or file I/O needed), implement `getNumPrograms()`/`setCurrentProgram()`/`getProgramName()`, and apply preset state by calling `apvts.replaceState()` — always from the message thread. For user-saved presets, the APVTS state XML is written/read from disk on the message thread only.

**Primary recommendation:** Implement factory presets as embedded XML strings in `plugin/Presets.h`, load via `setCurrentProgram()` calling `apvts.replaceState()` on the message thread. The existing state save/restore is already correct — do not change it.

### Pattern 1: Factory Preset as Embedded XML String

**What:** Each factory preset is a hard-coded XML string (the same format APVTS writes) stored in `Presets.h`.
**When to use:** Small number of factory presets (5-20). No file I/O, no BinaryData, no CMake changes.

```cpp
// Source: JUCE forum "Hard coded presets" thread
// plugin/Presets.h
#pragma once
#include <juce_core/juce_core.h>
#include "ParamIDs.h"

namespace XYZPresets {

struct Preset {
    const char* name;
    const char* xml;  // Full APVTS state XML
};

// State XML can be generated by running the plugin, configuring the desired
// state, and calling apvts.copyState().createXml()->toString() from a debug menu.
// Or hand-authored by matching the APVTS ValueTree format:
// <XYZPanState>
//   <PARAM id="x" value="0.0"/>
//   <PARAM id="y" value="1.0"/>
//   ...
// </XYZPanState>

static constexpr const char* kDefaultXml = R"(
<XYZPanState>
  <PARAM id="x" value="0.0"/>
  <PARAM id="y" value="1.0"/>
  <PARAM id="z" value="0.0"/>
  <PARAM id="r" value="1.0"/>
  <PARAM id="lfo_x_depth" value="0.0"/>
  <PARAM id="lfo_y_depth" value="0.0"/>
  <PARAM id="lfo_z_depth" value="0.0"/>
  <PARAM id="verb_wet" value="0.3"/>
</XYZPanState>
)";

// ... more presets ...

static const Preset kFactoryPresets[] = {
    { "Default",          kDefaultXml  },
    { "Orbit XY",         kOrbitXYXml  },
    { "Slow Drift",       kSlowDriftXml },
    { "Fly Around",       kFlyAroundXml },
    { "Deep Space",       kDeepSpaceXml },
};

static constexpr int kNumPresets = (int)(sizeof(kFactoryPresets) / sizeof(kFactoryPresets[0]));

} // namespace XYZPresets
```

### Pattern 2: setCurrentProgram — Load Preset on Message Thread

**What:** `setCurrentProgram()` parses XML and calls `apvts.replaceState()`. This is called by the DAW when the user picks a preset from the DAW's preset list.
**Critical:** `replaceState()` must NOT be called from processBlock. It is only safe when not called from the realtime audio thread.

```cpp
// Source: JUCE docs + forum "Hard coded presets" + VanillaJuce example
// In PluginProcessor.cpp

int XYZPanProcessor::getNumPrograms() override {
    return XYZPresets::kNumPresets;
}

int XYZPanProcessor::getCurrentProgram() override {
    return currentPresetIndex_;  // private int member, default 0
}

const juce::String XYZPanProcessor::getProgramName(int index) override {
    if (index >= 0 && index < XYZPresets::kNumPresets)
        return XYZPresets::kFactoryPresets[index].name;
    return "Unknown";
}

void XYZPanProcessor::setCurrentProgram(int index) {
    if (index < 0 || index >= XYZPresets::kNumPresets)
        return;

    currentPresetIndex_ = index;

    // Parse XML and replace APVTS state.
    // replaceState() is thread-safe (uses internal locks) but NOT realtime-safe.
    // setCurrentProgram is called from the message thread in all major DAWs.
    // If called from a background thread (rare), replaceState() is still safe
    // because the audio thread only reads the atomic parameter values, not the
    // ValueTree structure itself.
    auto xml = juce::parseXML(XYZPresets::kFactoryPresets[index].xml);
    if (xml != nullptr && xml->hasTagName(apvts.state.getType()))
        apvts.replaceState(juce::ValueTree::fromXml(*xml));
}
```

### Pattern 3: State Save/Restore (already implemented, verified correct)

```cpp
// Source: JUCE tutorial "Saving and loading your plug-in state"
// Already in PluginProcessor.cpp — confirmed correct, DO NOT CHANGE

void XYZPanProcessor::getStateInformation(juce::MemoryBlock& destData) {
    auto state = apvts.copyState();
    std::unique_ptr<juce::XmlElement> xml(state.createXml());
    copyXmlToBinary(*xml, destData);

void XYZPanProcessor::setStateInformation(const void* data, int sizeInBytes) {
    std::unique_ptr<juce::XmlElement> xml(getXmlFromBinary(data, sizeInBytes));
    if (xml != nullptr && xml->hasTagName(apvts.state.getType()))
        apvts.replaceState(juce::ValueTree::fromXml(*xml));
}
```

### Pattern 4: Defensive setStateInformation for background-thread hosts

Some hosts (GarageBand iOS AUv3, some background loaders) call `setStateInformation` from a non-message thread. The pattern below is defensive but adds complexity. **Verdict for XYZPan v1:** The existing `replaceState()` call is already thread-safe per JUCE docs. The atomic reads in `processBlock()` do not take any lock that `replaceState()` also holds. This extra pattern is NOT needed unless pluginval or a specific DAW causes failures.

```cpp
// OPTIONAL: defensive pattern — defer to message thread
// Only needed if replaceState causes issues in specific hosts
void XYZPanProcessor::setStateInformation(const void* data, int sizeInBytes) {
    // Parse synchronously (safe on any thread)
    auto xmlState = juce::MemoryBlock(data, sizeInBytes);
    juce::MessageManager::callAsync([this, xmlState = std::move(xmlState)]() {
        std::unique_ptr<juce::XmlElement> xml(
            getXmlFromBinary(xmlState.getData(), (int)xmlState.getSize()));
        if (xml != nullptr && xml->hasTagName(apvts.state.getType()))
            apvts.replaceState(juce::ValueTree::fromXml(*xml));
    });
}
// WARNING: callAsync version will fail pluginval "state restored" check if pluginval
// reads state back synchronously after calling setStateInformation.
// Start with the direct replaceState() version. Only switch to callAsync if
// a specific host crash is observed.
```

### Anti-Patterns to Avoid

- **Calling `replaceState()` from `processBlock()`:** replaceState uses locks; the audio thread must never block. This is almost certainly what caused the previous segfault.
- **Calling `apvts.state.setProperty()` directly from the audio thread:** ValueTree operations are NOT atomic and NOT realtime-safe.
- **Allocating `std::vector` or `std::string` inside `processBlock()`:** Even temporary XML parsing triggered by a flag in processBlock would allocate heap memory.
- **Storing a parsed `juce::XmlElement` in a `std::atomic<XmlElement*>` and accessing it on the audio thread:** XML objects are heap-allocated; accessing them on the audio thread is unsafe.
- **Calling `apvts.getRawParameterValue()` lookup (by string) on the audio thread:** The lookup itself is not realtime-safe. Pointer caching in the constructor (already done) is the correct pattern.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XML serialization of parameter state | Custom binary format or JSON serializer | APVTS copyState/replaceState + ValueTree XML | Already handles parameter ID mapping, missing params (uses defaults), version skew |
| Thread-safe parameter update from UI | Manual atomic arrays | APVTS SliderAttachment/ButtonAttachment | Handles notification, undo, host automation |
| Preset name list | Custom string array class | `const Preset[]` array + `getNumPrograms()`/`getProgramName()` | DAW interop requires the standard API |

**Key insight:** JUCE APVTS already solves the hard problem of mapping between ValueTree XML and `std::atomic<float>*` parameter values. The atomics are updated by JUCE internals when `replaceState()` is called. The audio thread reads via `.load()` on the already-cached atomic pointers — this path is entirely lock-free and allocation-free.

## Common Pitfalls

### Pitfall 1: replaceState on the audio thread (CRITICAL — previous segfault cause)
**What goes wrong:** `apvts.replaceState()` acquires an internal `juce::CriticalSection` lock. If called from `processBlock()`, this blocks the realtime audio thread, causing priority inversion, random stalls, and eventual heap corruption or crash.
**Why it happens:** A preset-loading flag is set by the UI, checked in `processBlock()`, and state is applied there.
**How to avoid:** NEVER call `replaceState()`, `copyState()`, `apvts.state.setProperty()`, or any `ValueTree` method from `processBlock()`. All preset application happens from `setCurrentProgram()` or `setStateInformation()`, which are called from non-audio threads.
**Warning signs:** Plugin runs fine for seconds or minutes then crashes randomly with no input. Crash is inside a lock acquisition or heap allocation deep in JUCE internals.

### Pitfall 2: XML tag name mismatch on reload
**What goes wrong:** Saved state has tag `"XYZPanState"` but replaceState check uses `apvts.state.getType()` which returns a `juce::Identifier`. If the state was saved under a different root name the check fails silently and state is not restored.
**Why it happens:** APVTS root name is set in constructor: `apvts(*this, nullptr, "XYZPanState", ...)`. The string must match in both save and restore.
**How to avoid:** Always use `xml->hasTagName(apvts.state.getType())` for the check, which uses the live APVTS root name. Never hardcode `"XYZPanState"` as a string literal in the check.
**Warning signs:** State not restored after DAW project reload; DAW reports preset changed but all knobs reset to default.

### Pitfall 3: setCurrentProgram not called by VST3 hosts
**What goes wrong:** `setCurrentProgram()` is never invoked by some VST3 hosts (confirmed: Reaper on Linux, some versions of FL Studio). The preset exists in `getNumPrograms()` but cannot be recalled via the host's preset list.
**Why it happens:** VST3 spec technically maps program changes differently than VST2. JUCE implements a workaround but it is not universally effective.
**How to avoid:** For v1, provide factory preset recall from a UI button or dropdown within the plugin's own editor (not relying on the host's preset list). The host-level preset list is a best-effort bonus. The critical requirement (PARAM-04) is state save/restore across sessions, not in-host preset browsing.
**Warning signs:** Presets show in DAW list but clicking them does nothing; getCurrentProgram() always returns 0.

### Pitfall 4: Partial XML preset (missing parameter IDs)
**What goes wrong:** A factory preset XML only includes the "interesting" parameters (X, Y, Z, LFO). When `replaceState()` is applied, the APVTS only updates parameters present in the XML; parameters absent from the XML keep their previous values. This is actually intended JUCE behavior but can cause unexpected state bleed.
**Why it happens:** Factory preset XML was hand-authored without listing all parameters.
**How to avoid:** Factory preset XMLs should include ALL parameter IDs, especially any that should reset to default (e.g. `verb_wet`, dev panel tunings). Generate preset XMLs by running the plugin and capturing `apvts.copyState().createXml()->toString()` from a debug output.
**Warning signs:** Loading a preset leaves reverb wet from the previous preset; dev panel values bleed between presets.

### Pitfall 5: currentPresetIndex_ not persisted in state
**What goes wrong:** `getCurrentProgram()` returns 0 after reload because `currentPresetIndex_` is not saved in `getStateInformation`.
**Why it happens:** APVTS XML only saves parameter values, not the `currentPresetIndex_` member. The int must be separately saved as a non-parameter property on the APVTS ValueTree state, or as a separate attribute in the XML.
**How to avoid:** Either (a) save `currentPresetIndex_` as a property on `apvts.state` before calling `copyState()`, or (b) accept that the DAW's preset index is not persisted (the parameter values themselves ARE correctly persisted via APVTS, which is what matters).
**Warning signs:** After DAW project reload, preset selector shows "Default" even though a different preset was active.

### Pitfall 6: AudioParameterChoice beat-div params not in simple XML format
**What goes wrong:** `AudioParameterChoice` parameters (LFO_X_BEAT_DIV, etc.) serialize as a choice index (0-10), not a float value. When hand-authoring preset XML, writing `value="0.5"` fails — the correct value for the default beat div (index 5 = "1/4") is `value="5"`.
**Why it happens:** Choice parameters store their integer index, not the float multiplier that the engine uses.
**How to avoid:** Generate preset XMLs from the running plugin, not by hand. If hand-authoring, check `kBeatDivDefaultIndex` from Constants.h for the correct index value.

```cpp
// Source: JUCE forum pattern for preset round-trip testing
TEST_CASE("Preset round-trip: X/Y/Z values restored", "[presets][PARAM-04]") {
    XYZPanProcessor proc;

    // Set non-default values
    proc.apvts.getParameter(ParamID::X)->setValueNotifyingHost(
        proc.apvts.getParameterRange(ParamID::X).convertTo0to1(0.5f));
    proc.apvts.getParameter(ParamID::LFO_X_DEPTH)->setValueNotifyingHost(
        proc.apvts.getParameterRange(ParamID::LFO_X_DEPTH).convertTo0to1(0.75f));

    // Save state
    juce::MemoryBlock block;
    proc.getStateInformation(block);

    // Reset to default
    proc.apvts.getParameter(ParamID::X)->setValueNotifyingHost(
        proc.apvts.getParameter(ParamID::X)->getDefaultValue());

    // Restore state
    proc.setStateInformation(block.getData(), (int)block.getSize());

    // Verify
    auto xRange = proc.apvts.getParameterRange(ParamID::X);
    float xVal = xRange.convertFrom0to1(proc.apvts.getParameter(ParamID::X)->getValue());
    REQUIRE(xVal == Catch::Approx(0.5f).margin(0.01f));
}
```

## Open Questions

1. **Should user-saveable presets (file-based) be in v1 scope?**
   - What we know: PARAM-05 says "factory presets demonstrating spatial positions and LFO patterns" — does not mention user-save
   - What's unclear: The phase description says "user-saveable presets" — but REQUIREMENTS.md PARAM-05 only mentions factory presets; EXT-04 ("Preset browser with categories") is v2
   - Recommendation: Implement factory presets only. User-save is file I/O that needs a UI (FileChooser dialog), is not in PARAM-05, and adds scope. Defer to EXT-04.

2. **How many factory presets and what should they demonstrate?**
   - What we know: Needs to show "spatial positions and LFO patterns" per PARAM-05
   - Recommendation: 5-8 presets: Default (silence / Y=1 position), Orbit XY (LFO on X+Y with sine), Slow Drift (low-rate X LFO), Fly Around (3-axis LFO), Near Far (Z modulation), Stereo Spread (stereo width), Hall Reverb (reverb-heavy static position)

3. **Will pluginval flag the callAsync deferred setStateInformation?**
   - What we know: Some forum reports indicate pluginval checks state restoration synchronously — callAsync would fail this check
   - Recommendation: Use direct `replaceState()` (already in codebase). Only switch to callAsync if a specific DAW crash is observed. The direct version is correct for all standard hosts.

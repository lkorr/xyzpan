# Phase 07.5: VST UI Design Overhaul - Research

**Researched:** 2026-03-20
**Domain:** JUCE plugin UI layout, custom LookAndFeel, OpenGL 3D view integration, VST plugin UX patterns
**Confidence:** HIGH (full codebase audit — all UI source files read directly)

---

## Summary

Phase 07.5 is an inserted urgent phase with no pre-defined requirements. The scope must be derived from a direct audit of the existing UI code. After reading every UI file, the current plugin has:

- A functioning Alchemy theme (dark purple-black / bronze / gold palette) with a custom `AlchemyLookAndFeel` subclassing `LookAndFeel_V4`
- An OpenGL 3D spatial view (`XYZPanGLView`) that fills the center/right of the window
- A left column (400px) with X/Y/Z position knobs and 3 LFO strips
- A bottom row (240px) with stereo orbit controls + 3 orbit LFO strips + reverb knobs + DEV toggle
- A preset bar (32px) across the top
- Layout constants hard-coded to 1100×1100px default

The primary problems identified: the window is too tall (1100px is unusable on standard monitors), the LFO strip layout is cramped and hard to read, the bottom row packs too much content into 240px, the ToggleButton/ComboBox theming is incomplete (not styled by AlchemyLookAndFeel), and the overall vertical proportions of panels squeeze the OpenGL view. The "overhaul" is most likely about: sizing/proportions, layout reorganization, LFO strip UX clarity, and visual polish — not a ground-up redesign.

**Primary recommendation:** Resize the default window to ~1100×750px (landscape proportion), rebalance panel heights, fully theme ToggleButton widgets, and tighten the LFO strip layout so controls are accessible without the window needing to be enormous.

## Current State Audit

### Window and Layout Constants

From `PluginEditor.h`:
```
kDefaultW      = 1100   kDefaultH      = 1100  (square — bad on typical monitors)
kLeftColW      = 400    kBottomH       = 240   kPresetBarH    = 32
kSectionHdrH   = 24     kPadding       = 6
kOrbitCtrlW    = 240    kReverbColW    = 56    kReverbKnobSz  = 48
kMiscSectionH  = 80
setResizeLimits(950, 750, 1800, 1600)
```

**Problem:** 1100×1100 is a square plugin window. Most DAWs sit behind a producer's arrangement view — screen real estate is at a premium. The minimum width (950) is reasonable but the default height (1100) means the plugin will be taller than many laptop screens (1080px displays are common). The aspect ratio should be landscape (wider than tall), not square.

### Left Column (400px wide)

Contains:
- POSITION header (24px)
- 3 sub-columns: X | Y | Z, each with a 108px knob + 16px label + LFO strip filling remaining space
- LFO Speed slider row (32px)
- UTILITIES header (24px) + Sphere Radius + Doppler knobs (80px)

**Problem:** At 1100px height with kBottomH=240 and kPresetBarH=32, the left column height is 828px. The position section minus utilities and headers is roughly 670px, so the LFO strips each get ~540px of height. That's enormous and mostly empty. If the window height drops to 750px, LFO strips shrink to ~240px — which actually looks good. The layout math is sound; the window height is just wrong.

**LFO strip internal layout (LFOStrip.cpp):**
```
Row 0: Shape selector  (24px)
Row 1: Waveform display (flexible, 30-80px cap)
Row 2: 4 knobs: Rate, Depth, Phase, Smooth (54+13=67px)
Row 3: Sync button (18px)
Fixed total: 109px + display (capped 80px) = 189px minimum usable
```

The waveform display is capped at 80px. In the current 1100px window, the strip gets ~540px, so it shows the display at 80px and leaves ~350px of dead space below the shape selector row. There is no dead-space — resized() fills LFO strips from `posSection.getY()` to `posSection.getBottom()`, and the layout fills ALL of it correctly. But since strip height is 540px and fixed content is only 189px, the content sits at the top and the bottom is background. This is not a JUCE bug — it's just that the window is too tall for the content quantity.

### Bottom Row (240px)

Contains in one horizontal band:
- STEREO ORBIT section: 240px orbit controls (width/phase/offset sliders + toggles) + 3 orbit LFO strips (flexible width)
- REVERB section: 4×56px=224px (Size/Decay/Damp/Wet rotary knobs)
- DEV toggle: 48px

**Problem:** 240px is tight for 3 LFO strips (shape selector + knobs + sync button = 109px minimum). The orbit LFO strips' waveform display is likely hidden (too short for kDisplayMinH=30) or barely visible. On top of that, the orbit controls section has 4 linear sliders stacked — reading Width/Phase/Offset/Speed as small horizontal sliders is unintuitive.

### ToggleButton and ComboBox Theming

`AlchemyLookAndFeel` implements:
- `drawRotarySlider` (custom gold arc)
- `drawLinearSlider` (custom hero/normal modes)
- `drawButtonBackground` (applies to `TextButton` and `ToggleButton`)
- `drawButtonText` (applies to `TextButton`)
- `drawLabel`

**Gap:** `ToggleButton` does not have `drawToggleButton` override. JUCE `LookAndFeel_V4::drawToggleButton` draws a standard checkbox tick box. The Alchemy theme only partially covers toggle buttons through `drawButtonBackground` (fill + bronze border), but the tick mark itself is drawn by the default implementation — using JUCE's standard colors which may not match the palette.

The `faceListenerToggle_` and `orbitTempoSyncToggle_` in the editor use `ToggleButton` with `setClickingTogglesState(true)` but styled text. These use `drawButtonBackground` + `drawButtonText` from LookAndFeel_V4 (not a custom override) — meaning the tick mark state indicator defaults to JUCE's standard style.

**ComboBox:** `AlchemyLookAndFeel` sets `ComboBox::backgroundColourId` and `ComboBox::textColourId` but does not override `drawComboBox`. The arrow dropdown indicator and popup menu styling are partially covered by color settings but the actual border/arrow geometry comes from LookAndFeel_V4 default.

### LFO Strip Density

6 LFO strips total (X, Y, Z + orbitXY, orbitXZ, orbitYZ). Each strip has:
- LFOShapeSelector: 6 mini waveform-preview buttons in a row (18px tall)
- LFOWaveformDisplay: live oscilloscope (0-80px)
- 4 knobs: Rate/Depth/Phase/Smooth (54px + 13px label = 67px)
- Sync button (18px)

The orbit LFO strips sit in a bottom row with only ~154px available for content (240px minus section header). With 3 strips squeezed horizontally, each strip is `(totalW - 240) / 3` wide. At 1100px total width, orbitTotalW ≈ 588px, so each strip is ~196px wide. That width is fine for 4 knobs at 54px each but the height constraint (154px usable) is the issue — waveform display will be hidden, sync button may be cut off.

### Missing LookAndFeel Overrides

The following are used in the plugin but not themed by `AlchemyLookAndFeel`:
- `drawToggleButton` — tick mark not styled
- `drawComboBox` — dropdown arrow/border uses LookAndFeel_V4 defaults
- `drawPopupMenuItem` — preset dropdown popup menu items
- `drawScrollbar` — dev panel viewport scrollbar
- `getSliderThumbRadius` — affects thumb hit area for linear sliders

### Snap Buttons

Three TextButtons ("XY", "XZ", "YZ") in the top-right corner of the GL view. These are 40×24px — small but functional. They do not toggle state (clicking snaps camera, no persistent on/off state). Visual feedback for "currently snapped to XY" vs free orbit is absent.

### Section Header Visual Quality

`drawHeader` lambda in `paint()` uses a gradient fill (Bronze→DarkIron left-to-right) with BrightGold label text and a Bronze underline. This is functional but the gradient is subtle. The "STEREO ORBIT" and "REVERB" headers share the bottom row — they are drawn correctly.

## Architecture Patterns

### JUCE LookAndFeel Override Pattern

To style ToggleButton tick marks in LookAndFeel_V4:
```cpp
// Source: JUCE LookAndFeel_V4 documentation / source
void drawToggleButton(juce::Graphics& g, juce::ToggleButton& button,
                      bool shouldDrawButtonAsHighlighted,
                      bool shouldDrawButtonAsDown) override
{
    // Option A: render as a styled TextButton with on/off color distinction
    // (best for the Alchemy theme where toggles are labeled buttons, not checkboxes)
    auto bounds = button.getLocalBounds().toFloat().reduced(0.5f);

    juce::Colour fill = juce::Colour(kDarkIron);
    if (shouldDrawButtonAsDown)  fill = fill.brighter(0.2f);
    if (shouldDrawButtonAsHighlighted) fill = fill.brighter(0.1f);
    g.setColour(fill);
    g.fillRoundedRectangle(bounds, 3.0f);

    // Border: warm gold when toggled on, bronze when off
    const juce::Colour border = button.getToggleState()
        ? juce::Colour(kWarmGold)
        : juce::Colour(kBronze);
    g.setColour(border);
    g.drawRoundedRectangle(bounds, 3.0f, 1.5f);

    // Text: bright gold when on, parchment when off
    g.setFont(juce::Font(juce::FontOptions(juce::jmin(12.0f,
        static_cast<float>(button.getHeight()) * 0.5f))));
    g.setColour(button.getToggleState()
        ? juce::Colour(kBrightGold)
        : juce::Colour(kParchment).withMultipliedAlpha(
            button.isEnabled() ? 1.0f : 0.5f));
    g.drawText(button.getButtonText(), button.getLocalBounds(),
               juce::Justification::centred, true);
}
```
**Confidence: HIGH** — JUCE LookAndFeel_V4 virtual method signatures confirmed from JUCE 8 source structure.

### Resized() Safety Patterns (JUCE CRITICAL)

When reworking `resized()`, always guard against zero-area rectangles:
```cpp
// Source: JUCE component layout best practice
auto area = getLocalBounds();
if (area.isEmpty()) return;  // guard before any removeFrom* calls

// Never pass a zero-height removeFromTop to child.setBounds
auto row = area.removeFromTop(kSomeHeight);
if (row.getHeight() > 0)
    child.setBounds(row);
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Custom font loading | Font file parsing | `juce::Font(juce::FontOptions(...))` | JUCE handles font subsystem |
| Custom scroll physics | Velocity tracking | `juce::Viewport` (already used in DevPanel) | Edge cases in scroll behavior |
| Custom popup menu | Manual overlay component | `juce::PopupMenu` styled via LookAndFeel | JUCE handles keyboard nav, dismiss |
| Custom ComboBox popup | Re-implement dropdown | Style via `drawComboBox` + `drawPopupMenuItem` | JUCE wires the data model |
| Custom ToggleButton | Reimplementing toggle | Override `drawToggleButton` | 3 lines vs 100 |

**Key insight:** Every custom UI widget in this codebase is already a JUCE component with a custom LookAndFeel draw override. Extend that pattern — don't introduce new widget base classes.

## Common Pitfalls

### Pitfall 1: setDefaultLookAndFeel vs setLookAndFeel
**What goes wrong:** `XYZPanEditor` calls both `setLookAndFeel(&lookAndFeel_)` AND `juce::LookAndFeel::setDefaultLookAndFeel(&lookAndFeel_)`. The second call sets the global default for all JUCE components in the process — including any other plugin instances and DAW UI. This is a known pitfall: multiple plugin instances fighting over the global LookAndFeel.
**How to avoid:** Only call `setLookAndFeel` on the editor component, not `setDefaultLookAndFeel`. The editor's children inherit their parent's LookAndFeel. If popup menus don't inherit correctly, call `setLookAndFeel` explicitly on the ComboBox or handle it in `lookAndFeel_changed()`.
**Warning signs:** Second instance of the plugin has broken appearance; DAW UI is affected after loading the plugin.
**Confidence: HIGH** — JUCE documentation explicitly warns against `setDefaultLookAndFeel` in plugin editors.

### Pitfall 2: ToggleButton Default Rendering
**What goes wrong:** `LookAndFeel_V4::drawToggleButton` draws a tick-box style control with a checkmark. If `AlchemyLookAndFeel` does not override `drawToggleButton`, `faceListenerToggle_` and `orbitTempoSyncToggle_` render with the V4 default (a box and a tick mark at left, text at right) even though `drawButtonBackground` IS overridden.
**Root cause:** `drawButtonBackground` is called by TextButton rendering but `drawToggleButton` is a separate virtual that calls its own background + tick logic in V4. The two are independent dispatch points.
**How to avoid:** Add `drawToggleButton` override to `AlchemyLookAndFeel`. Make it render like a TextButton (text-only, no tick box) with gold border when active.

### Pitfall 3: Resized() Calling During Construction
**What goes wrong:** If `resized()` is called before all member sliders/labels are constructed (e.g., during `setSize()` in constructor), any member it accesses must already exist. Current code is fine because `setSize()` is the last line of the constructor.
**How to avoid:** Keep `setSize()` as the last line of the constructor. Any new layout constants must be initialized before `resized()` runs.

### Pitfall 4: setDefaultLookAndFeel Destructor Race
**What goes wrong:** `XYZPanEditor::~XYZPanEditor()` calls `juce::LookAndFeel::setDefaultLookAndFeel(nullptr)` to restore the default, but if two instances exist and one closes, it nulls the global LookAndFeel for the other instance.
**How to avoid:** Remove the `setDefaultLookAndFeel` call entirely (from constructor and destructor). Use `setLookAndFeel` on the editor only. All child components added with `addAndMakeVisible` inherit the parent's LookAndFeel automatically.

### Pitfall 5: Paint Geometry Diverging From Resized Geometry
**What goes wrong:** `paint()` manually replicates layout math from `resized()` to draw backgrounds and dividers at the right positions. If `resized()` changes a layout constant, `paint()` must be updated too. Currently they share constants like `kBottomH`, `kPresetBarH`, `kSectionHdrH`, `kOrbitCtrlW`, `kReverbSectionW` — but the lambda `drawHeader` and the divider positions are computed independently in both methods.
**How to avoid:** Extract all geometry into a `computeLayout()` struct that both `paint()` and `resized()` call. This is the safe refactoring pattern.

### Pitfall 6: LFO Strip Sync Button Overflow
**What goes wrong:** The sync button (18px) is placed at `row3Y = row2Y + kKnobSz + kLabelH` — below the 4 knobs. If the strip height is less than `kShapeTopMargin + kShapeRowH + 2 + kKnobSz + kLabelH + kSyncH = 4+18+2+54+13+18 = 109px`, the sync button will overflow past the strip boundary and visually overlap adjacent content.
**How to avoid:** Always set LFO strip height ≥ 109px in `resized()`. With the new window height (750px), verify the per-strip height calculation produces ≥ 109px.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| LookAndFeel_V1/V2 default | LookAndFeel_V4 (base class) | JUCE 5+ | Flat design, easier to subclass |
| Per-component setColour() only | Override virtual draw methods | Always valid | Full control of geometry, not just color |
| setDefaultLookAndFeel in plugin editors | setLookAndFeel on editor only + child inheritance | Recognized pitfall | Avoids multi-instance LookAndFeel corruption |

**Deprecated patterns to avoid:**
- `juce::Font(12.0f)` — use `juce::Font(juce::FontOptions(12.0f))` (JUCE 8 constructor)
- `LookAndFeel::setDefaultLookAndFeel` in plugin editors — use `setLookAndFeel` instead
- `juce::Slider::setTextValueSuffix` with empty string to hide suffix — use `setNumDecimalPlacesToDisplay(0)` for integer params

## Open Questions

1. **What specifically prompted "UI Design Overhaul"?**
   - What we know: Phase was inserted as "URGENT" with no requirements defined — likely prompted by a real complaint about the plugin's visual appearance or usability in practice
   - What's unclear: Is there a specific screenshot or mockup the user has in mind? Is this about proportions, theming, or a fundamentally different layout?
   - Recommendation: Ask the user for specific complaints / desired outcome before locking a plan. The audit above identifies all auditable problems but the user may have a concrete vision.

2. **Should the bottom row become a resizable panel or stay fixed at 240px?**
   - What we know: kBottomH=240 is tight for 3 LFO strips (minimum 109px each) at the current 30fps waveform display minimum
   - What's unclear: User preference for bottom row height
   - Recommendation: Make it adjustable (225-300px range) or increase to 280px fixed

3. **Should orbit controls become rotary knobs to match the position section?**
   - What we know: Orbit Width/Phase/Offset/Speed are currently linear horizontal sliders (compact) while X/Y/Z are large rotary knobs
   - Recommendation: Keep as linear sliders in the bottom row — rotary knobs need more vertical space than the bottom row provides. Alternatively, move orbit controls to a collapsible section.

4. **Is the `setDefaultLookAndFeel` bug causing observable problems?**
   - What we know: It is called in constructor AND removed in destructor. With a single plugin instance this is harmless. With two instances the race condition exists.
   - Recommendation: Fix regardless — it's a correctness issue even if not yet triggered.

# Phase 07.2: Optimization Round 2 -- DSP and UI CPU Optimization - Research

**Researched:** 2026-03-17
**Domain:** Profile-driven CPU optimization of DSP engine hot paths and OpenGL UI rendering
**Confidence:** HIGH (all source code is fully visible; analysis is from direct code inspection)

---

## Summary

Phase 7.1 eliminated the biggest bottleneck: per-sample transcendental math in `setCoefficients()` calls (moved to per-block), added CMake Release flags (`/O2 /arch:AVX2 /fp:fast`), and moved the stereo orbit angular smoother to per-block. Those were the "obvious" wins.

Phase 7.2 must now target the **remaining per-sample cost** in the inner loop and the **OpenGL rendering overhead**. After thorough code inspection, the following optimization domains emerge:

1. **DSP inner loop**: The per-sample loop still contains ~15 `std::sqrt` calls, ~8 `std::cos`/`std::sin` calls (stereo orbit XZ/YZ per sample), `std::abs`, `std::clamp`, and branching on stereo mode. The LFO `tick()` calls `std::sin` per sample for Sine waveform. The FDN reverb runs 4 delay reads + Householder matrix per sample. The comb bank runs 10 serial feedback comb filters per sample.

2. **OpenGL rendering**: The GL view uses `setContinuousRepainting(true)` -- rendering at vsync rate (~60fps) regardless of whether anything changed. Each frame issues ~12-15 draw calls (room wireframe, floor grid, listener sphere, forward cone, 2 ears, audible radius sphere, source sphere, 2 stereo nodes, 3 trails). Uniform uploads happen per draw call with redundant `sphereShader_->use()` calls. The sphere mesh is 16x16 segments (578 vertices, 960 triangles) reused for all sphere draws.

3. **Plugin-level**: The `processBlock` parameter snapshot involves ~80 atomic loads, but these are cheap. The position bridge writes a `SourcePositionSnapshot` every block with a `std::sqrt` for distance. The DSP state bridge writes every block. These are minor but worth noting.

**Primary recommendation:** Focus on three high-impact areas: (1) LFO sine lookup table to eliminate per-sample `std::sin`, (2) reduce per-sample `std::sqrt` calls via block-rate position caching for the mono path, (3) switch GL view from continuous repainting to dirty-flag repainting. Profile before and after each change using the existing microbenchmark tests.

### Pattern 1: Sine Lookup Table for LFO

**What:** Replace `std::sin(accumulator_ * 2*PI)` in `LFO::tick()` with a table lookup + linear interpolation. The LFO runs at audio rate (44100-192000 Hz) and `std::sin` costs ~15-20ns per call. With 6 LFOs active (3 position + 3 orbit), that's 6 * 20ns = 120ns per sample wasted on transcendentals.

**Implementation:**
```cpp
// engine/include/xyzpan/dsp/SineLUT.h
#pragma once
#include <array>
#include <cmath>

namespace xyzpan::dsp {

class SineLUT {
public:
    static constexpr int kSize = 2048;  // 2048 entries = 11-bit precision

    static float lookup(float phase01) {
        // phase01 is in [0, 1)
        const float idx = phase01 * kSize;
        const int i0 = static_cast<int>(idx) & (kSize - 1);
        const int i1 = (i0 + 1) & (kSize - 1);
        const float frac = idx - static_cast<float>(static_cast<int>(idx));
        return table_[i0] + frac * (table_[i1] - table_[i0]);  // linear interp
    }

private:
    static inline const std::array<float, kSize> table_ = []() {
        std::array<float, kSize> t{};
        for (int i = 0; i < kSize; ++i)
            t[i] = std::sin(2.0f * 3.14159265358979f * static_cast<float>(i) / kSize);
        return t;
    }();
};

} // namespace xyzpan::dsp
```

**Cost reduction:** `std::sin` (~15-20ns) -> LUT lookup (~2-3ns). At 6 LFOs * 44100 Hz = 264600 lookups/sec, saves ~3-5us per 64-sample block.

**Accuracy:** 2048-point table with linear interpolation gives <0.001 error. For LFO modulation of position (range [-1,1]), this is inaudible. The LFO is not in the audio path -- it modulates position coordinates.

### Pattern 2: Reduce Per-Sample sqrt Calls

**What:** The per-sample loop contains multiple `std::sqrt` calls that could be reduced:

**Current per-sample sqrt inventory (mono path):**
1. `std::sqrt(modX*modX + modY*modY + modZ*modZ)` -- raw modulated distance (line ~983)
2. `std::sqrt(modX*modX + modY*modY)` -- horizontal magnitude (line ~993)
3. Inside `processBinauralForSource()`: `std::sqrt(nodeX*nodeX + nodeY*nodeY)` (line ~324)
4. Distance for DSP state: already uses computeDistance (another sqrt)

**Stereo path adds:**
5-6. `processDistanceForNode()` x2: each calls `std::sqrt(nodeX*nodeX + ...)` (line ~421)

That's 3-6 `std::sqrt` calls per sample (each ~8-12ns on modern x86 with `/fp:fast`).

**Optimization strategies:**
- **Mono path:** `horizontalMag` and `rawModDist` are computed from `modX/modY/modZ` which only change because LFOs tick. When LFO depths are zero, these are block-constant. Add a fast path: if all LFO depths are zero, compute once at block start.
- **Stereo path:** The L/R node positions change per sample due to orbit LFOs. The sqrt inside `processBinauralForSource()` and `processDistanceForNode()` cannot easily be hoisted. However, `nodeHorizMag` could use a fast inverse sqrt approximation for the normalization (only used to compute `nodeX / nodeHorizMag` for azimuth factor -- relative accuracy matters, not absolute).
- **Alternative:** Use `modX*modX + modY*modY` (squared distance) directly where possible. Several comparisons like `horizontalMag > 1e-7f` can use `horizMagSq > 1e-14f`. The division `nodeX / nodeHorizMag` still needs the sqrt.

**Practical recommendation:** The fast path for zero-LFO-depth is the highest value. When LFOs are inactive (common use case), all position-derived values are block-constant and the entire per-sample position computation can be skipped.

### Pattern 3: GL View Dirty-Flag Repainting

**What:** `glContext_.setContinuousRepainting(true)` causes the GL view to render at vsync rate (typically 60fps) even when nothing has changed (no parameter movement, no camera orbit). This wastes CPU/GPU resources.

**Optimization:**
```cpp
// In constructor:
glContext_.setContinuousRepainting(false);  // Don't render when idle

// In renderOpenGL():
// Check if anything changed since last frame
bool dirty = bridge_.hasNewData()  // Position changed
           || camera_.isDirty()     // Camera orbited
           || isDraggingSource_;    // User is dragging
if (!dirty) return;  // Skip rendering entirely

// ... existing render code ...
```

**Caveat:** JUCE's `setContinuousRepainting(false)` means `renderOpenGL()` is only called when `repaint()` is explicitly triggered. This requires:
1. A timer (e.g., 30Hz) that checks the bridge for new data and calls `repaint()` if changed.
2. Mouse events (drag, wheel) call `repaint()`.
3. Snap view changes call `repaint()`.

**Alternative approach (simpler):** Keep continuous repainting but add a frame rate limiter:
```cpp
void XYZPanGLView::renderOpenGL() {
    // Throttle to 30fps when idle (position not moving)
    const double now = juce::Time::getMillisecondCounterHiRes();
    const bool positionChanged = bridge_.hasNewData();
    const double minInterval = positionChanged ? 16.67 : 33.33;  // 60fps vs 30fps
    if (now - lastRenderTime_ < minInterval) return;
    lastRenderTime_ = now;
    // ... render ...
}
```

This is safer because it doesn't change the repaint model. When idle, it renders at 30fps instead of 60fps, cutting GL thread CPU by ~50%.

### Pattern 4: Draw Call Reduction via Uniform Caching

**What:** Each `drawSphere()` and `drawSphereWithModel()` call:
1. Calls `sphereShader_->use()` (program bind)
2. Uploads 3 mat4 uniforms (projection, view, model)
3. Uploads 3 scalar uniforms (color, opacity, lightDir)
4. Binds VAO
5. Calls `glDrawElements`

With ~8 sphere/cone draws per frame, that's 8 shader binds and 48 uniform uploads. The `projection` and `view` matrices are identical across all calls. `lightDir` is also constant.

**Optimization:** Bind shader once, upload shared uniforms once, then loop over instances:
```cpp
// In renderOpenGL(), before sphere draws:
sphereShader_->use();
sphereShader_->setUniformMat4("projection", ...);
sphereShader_->setUniformMat4("view", ...);
sphereShader_->setUniform("lightDir", ...);
glBindVertexArray(vaoSphere_);

// Each sphere draw only sets per-instance uniforms:
sphereShader_->setUniformMat4("model", ...);
sphereShader_->setUniform("nodeColor", ...);
sphereShader_->setUniform("opacity", ...);
glDrawElements(GL_TRIANGLES, sphereIndexCount_, GL_UNSIGNED_INT, nullptr);
```

This reduces shader binds from ~8 to 2 (sphere + line) and uniform uploads from ~48 to ~24.

### Pattern 5: Per-Sample Branch Elimination in Stereo Path

**What:** The inner loop has this branch structure:
```cpp
if (stereoActive) {
    // ~180 lines of stereo processing
} else {
    // ~90 lines of mono processing
}
```

`stereoActive` is constant across the block (it's based on `smoothedWidth > 0.001f` which is smoothed). The compiler cannot easily split this into two optimized loops because both paths share the same loop body with the shared pipeline afterward.

**Optimization:** Split the sample loop into two separate functions -- one for mono, one for stereo. Call the appropriate one based on the block-level `stereoActive` flag. This allows the compiler to optimize each loop independently (better register allocation, no dead-code speculation).

```cpp
if (stereoActive)
    processBlockStereo(inputs, outL, outR, auxL, auxR, numSamples, ...precomputed...);
else
    processBlockMono(inputs, outL, outR, auxL, auxR, numSamples, ...precomputed...);
```

**Value:** Moderate. The compiler may already optimize this well with branch prediction. Profile first.

### Pattern 6: FDN Reverb Batch Processing

**What:** `FDNReverb::processSample()` is called once per sample. It does:
- 1 delay push (pre-delay)
- 1 delay read (pre-delay)
- 4 delay reads (FDN lines)
- Householder matrix (4 adds + 4 multiply-subtract)
- 4 one-pole damping filters
- 4 delay pushes

That's 5 delay reads + 5 delay pushes + 12 multiplies per sample. At 44.1kHz this is ~44100 calls/sec.

**Optimization opportunity:** Process the FDN in blocks. Pre-read all delay outputs into small arrays, process the feedback matrix as a batch, then push all results back. This enables:
- Better cache locality (read all delays, then write all delays)
- Potential SIMD on the 4-element Householder matrix

**Cost estimate:** ~20-30ns per sample for FDN. Block processing might save 5-10ns via cache effects.

**Practical recommendation:** LOW priority. The FDN is already efficient. Only pursue if profiling shows it's a hot spot after other optimizations.

### Anti-Patterns to Avoid

- **Premature SIMD:** Do not write manual AVX2 intrinsics before profiling shows a specific loop is the bottleneck. The compiler's auto-vectorizer with `/O2 /arch:AVX2 /fp:fast` handles many cases.
- **Approximating audible DSP:** The sine LUT is safe for LFOs (position modulation) but must NOT be used for filter coefficients or audio-path processing where precision matters.
- **Breaking the zero-allocation contract:** All LUTs and caches must be allocated in `prepare()` or statically. No `new` or `std::vector` resize in `process()`.
- **Over-optimizing the UI thread:** The GL view is not on the audio thread. Optimization here improves user experience (lower CPU, better responsiveness) but does not affect audio quality. Don't sacrifice code clarity for marginal GL gains.
- **Changing DSP behavior:** All optimizations must be output-equivalent. The existing performance tests verify throughput; the existing audio correctness tests verify output. Both must pass.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sine approximation | Custom polynomial/CORDIC | 2048-point LUT with linear interp | Simpler, sufficient accuracy for LFO, no error analysis needed |
| Fast inverse sqrt | Custom Quake-style `rsqrt` | Let compiler use `_mm_rsqrt_ss` via `/fp:fast /arch:AVX2` | Compiler generates optimal SIMD rsqrt with `/fp:fast` |
| GPU instancing | Custom instance buffer | Per-instance uniform upload | Only 8 sphere draws -- instancing overhead isn't worth it for <10 instances |
| Profiling framework | Custom timing wrapper | `std::chrono::steady_clock` + existing TestPerformance.cpp | Already works, already in CI |

**Key insight:** The biggest remaining wins are from **avoiding work entirely** (zero-LFO fast path, dirty-flag GL repainting) rather than making existing work faster. Algorithmic shortcuts beat micro-optimization.

## Common Pitfalls

### Pitfall 1: Sine LUT Phase Discontinuity

**What goes wrong:** The LUT assumes `phase01` is in `[0, 1)`. If `accumulator_` drifts outside this range due to floating-point error, the lookup wraps incorrectly.
**Why it happens:** The LFO accumulator uses `if (accumulator_ >= 1.0f) accumulator_ -= 1.0f` which can leave tiny negative values after subtraction.
**How to avoid:** Use bitmask wrapping on the integer index: `& (kSize - 1)`. This is already shown in the Pattern 1 code. The fractional part is always valid because `static_cast<int>` truncates toward zero.
**Warning signs:** Occasional clicks or pops in LFO output at phase wrap points.

### Pitfall 2: Zero-LFO Fast Path Breaks Smoother State

**What goes wrong:** Skipping the per-sample position computation when LFO depths are zero means smoothers (`lfoDepthXSmooth_` etc.) are not ticked. When LFO depth later becomes non-zero, the smoother state is stale, causing a sudden jump.
**Why it happens:** `OnePoleSmooth::process(target)` must be called every sample to maintain state continuity.
**How to avoid:** The fast path must still tick all smoothers with their current (zero) targets. The optimization is to skip the sqrt/trig math that depends on LFO output, not the smoothers themselves. Specifically: when all LFO depths are zero, `modX = currentParams.x`, `modY = currentParams.y`, `modZ = currentParams.z` -- use the pre-computed block-start values for all position-derived quantities without recomputing sqrt.
**Warning signs:** Audible click when enabling an LFO after it was off for a while.

### Pitfall 3: GL Dirty Flag Misses State Changes

**What goes wrong:** With dirty-flag repainting, the GL view stops updating but the audio is still processing. If a parameter changes via automation (not mouse drag), the GL view doesn't know to repaint.
**Why it happens:** The bridge is updated every audio block, but if the GL thread never checks the bridge (because it's not repainting), it misses updates.
**How to avoid:** Use a timer-based approach: a 30Hz timer always checks the bridge and calls `repaint()` if data changed. This guarantees the view updates within 33ms of any parameter change. The timer is on the message thread (not GL thread) and is very cheap.
**Warning signs:** GL view freezes when automation moves parameters without mouse interaction.

### Pitfall 4: Stereo Orbit Per-Sample cos/sin Still Present

**What goes wrong:** Phase 7.1 moved the phase/offset angular smoother to per-block. But the orbit XZ and YZ plane rotations still call `std::cos(angXZ)` and `std::sin(angXZ)` per sample when those orbit depths are non-zero (lines 1079-1108 in Engine.cpp). These are guarded by `if (std::abs(orbitDepXZ) > 1e-7f)` but when active, it's 4 trig calls per sample.
**Impact:** When all 3 orbit planes are active, that's 4 cos + 4 sin = 8 trig calls per sample in the stereo path. At ~15ns each = 120ns/sample.
**How to handle:** These could use the same SineLUT (and cosine = sin(x + 0.25)). The orbit angle is `orbitRawXZ * orbitDepXZ * PI` -- convert to [0,1) phase and look up. However, the orbit LFO raw output is in [-1,1], so the angle is in [-PI, PI]. Need to normalize to [0,1) first: `phase01 = (angle / (2*PI)) + 0.5f; phase01 -= floor(phase01)`.
**Recommendation:** Include this in the SineLUT optimization task. Extend the LUT to support arbitrary angle inputs, not just [0,1) phase.

### Pitfall 5: INFRA-03 Regression from Cached Values

**What goes wrong:** If a cached value (e.g., a pre-computed LUT index, a block-start position) uses the old sample rate after `prepare()` is called with a new rate.
**How to avoid:** All per-block pre-computation derives from `sr = static_cast<float>(sampleRate)` at the top of `process()`. Any LUT that depends on sample rate must be rebuilt in `prepare()`. The sine LUT is sample-rate-independent (it's a pure math function), so it's safe. Filter coefficients already use `sr` per-block.
**Warning signs:** Pitch or timing artifacts when changing sample rate mid-session.

### Zero-LFO Fast Path in process()

```cpp
// After LFO depth smoothing, before position computation:
const bool lfoActive = depthX > 1e-7f || depthY > 1e-7f || depthZ > 1e-7f;

float modX, modY, modZ;
if (lfoActive) {
    modX = currentParams.x + lfoX_.tick() * depthX;
    modY = currentParams.y + lfoY_.tick() * depthY;
    modZ = currentParams.z + lfoZ_.tick() * depthZ;
} else {
    // Skip LFO tick but still advance accumulator to keep phase consistent
    lfoX_.tick(); lfoY_.tick(); lfoZ_.tick();
    // Use block-start values (no sqrt needed -- already computed in preamble)
    modX = blkX; modY = blkY; modZ = blkZ;
}

// When !lfoActive, skip per-sample sqrt for rawModDist, horizontalMag
// -- use blkDist, blkHorizMag from the per-block preamble
const float rawModDist = lfoActive
    ? std::sqrt(modX*modX + modY*modY + modZ*modZ)
    : blkDist;  // pre-computed before the loop
```

## OpenGL Rendering Cost Analysis

**Per-frame cost breakdown:**

| Operation | Count | Est. cost | Notes |
|-----------|-------|-----------|-------|
| Clear + viewport setup | 1 | ~5us | Minimal |
| Bridge read (atomic load) | 1 | ~100ns | Lock-free |
| Trail buffer management | 3 | ~2us | CPU-side circular buffer |
| Line shader draws (room + grid) | 2 | ~20us | ~200 vertices total |
| Sphere shader draws | 8 | ~80us | 960 triangles each, 8 binds |
| Trail shader draws | 3 | ~15us | Small vertex uploads |
| Uniform uploads | ~48 | ~24us | Redundant projection/view |
| **Total per frame** | | **~146us** | |

At 60fps: 146us * 60 = ~8.8ms/sec = ~0.9% CPU. Not huge, but:
- When the plugin window is in the background, 60fps rendering is pure waste
- On laptops, GPU wakeups at 60fps prevent power saving
- Reducing to 30fps when idle saves ~4.4ms/sec

**Highest-value GL optimization:** Throttle frame rate when position is stationary. This is a user-experience win (less CPU, better battery) more than a performance-critical fix.

## Optimization Priority Order

1. **Sine LUT for LFO** -- HIGH value, LOW risk. Eliminates ~108ns/sample with 6 active sine LFOs. Simple table lookup, no behavioral change. Affects both position and orbit LFOs.

2. **Zero-LFO fast path** -- HIGH value, MEDIUM risk. Skip per-sample sqrt when LFOs inactive. Most users run the plugin in mixing (no LFOs) more than in creative (LFOs active). Saves ~30ns/sample in the common case. Must preserve smoother state.

3. **GL view dirty-flag / throttled repainting** -- MEDIUM value, LOW risk. Reduces GL thread CPU by 30-50% when idle. No effect on audio quality. Improves laptop battery life.

4. **GL draw call batching** -- LOW value, LOW risk. Reduces uniform uploads from ~48 to ~24 per frame. Saves ~12us per frame. Cleaner code as a bonus.

5. **Orbit XZ/YZ cos/sin via LUT** -- MEDIUM value, LOW risk. Only matters when stereo orbit is active with XZ/YZ planes. Saves ~120ns/sample in that case. Can share the SineLUT infrastructure from item 1.

6. **Split mono/stereo loop** -- LOW value, MEDIUM risk. Compiler likely handles the branch well already. Profile before attempting. Adds code duplication.

7. **FDN batch processing** -- LOW value, HIGH effort. The reverb is already efficient. Only pursue if profiling shows it's a hot spot.

## Open Questions

1. **What is the actual measured per-block CPU time after Phase 7.1?**
   - What we know: Phase 7.1 eliminated per-sample setCoefficients and added compiler flags. Estimated ~300ns/sample post-optimization.
   - What's unclear: No actual measurement has been reported. The performance tests have budgets but we don't know current margins.
   - Recommendation: Run `ctest --test-dir build -R TestPerformance -C Release` and capture actual timings before any Phase 7.2 work. This establishes the baseline.

2. **Does the compiler auto-vectorize the filter chain?**
   - What we know: `/O2 /arch:AVX2 /fp:fast` is set. Filter `process()` functions are simple arithmetic (multiply-add).
   - What's unclear: The filters have state dependencies (each output depends on previous output via `ic1eq_`, `ic2eq_`, `z_`). This typically prevents auto-vectorization of sequential filter chains.
   - Recommendation: Use `/Qvec-report:2` to check. If the main loop is not auto-vectorized (likely), accept it -- IIR filters with state feedback fundamentally cannot be vectorized across samples. They CAN be vectorized across channels (L+R simultaneously) but this requires restructuring to SoA (struct-of-arrays) which is a significant refactor.

3. **Is the JUCE OpenGLContext timer-based repaint reliable on Windows?**
   - What we know: `setContinuousRepainting(false)` requires explicit `repaint()` calls. JUCE handles the scheduling.
   - What's unclear: Whether timer-triggered repaints from the message thread reach the GL thread reliably, or if there are latency/stutter issues.
   - Recommendation: Test both approaches (dirty flag + timer, and simple frame-rate throttle). The throttle approach is safer and simpler.

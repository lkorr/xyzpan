# Phase 7.1: Optimization - Research

**Researched:** 2026-03-14
**Domain:** Real-time DSP audio-thread CPU optimization — C++/MSVC, hand-rolled filter/delay primitives
**Confidence:** HIGH (code is fully visible; no external APIs to verify)

---

## Summary

Phase 7.1 targets the audio thread's per-sample processing loop in `XYZPanEngine::process()`. The
chain is computationally deep: every sample processes through 10 series comb filters, 6 biquads
(mono EQ), 4 SVF lowpass filters, 2–4 fractional delay reads (Hermite), air absorption LPFs, FDN
reverb, plus 3–6 LFO ticks and all associated smoothers. In stereo-width mode the binaural +
distance pipeline doubles.

The critical bottleneck areas visible from the source code are:

1. **BiquadFilter::setCoefficients() called per-sample** — `processBinauralForSource()` calls
   `setCoefficients()` on 6 biquads (presShelf, earCanal, P1, N1, N2, pShelf) and 4 near-field
   biquad calls every single sample, despite each containing `std::cos`, `std::sin`, `std::pow`,
   `std::sqrt`. The comment in `BiquadFilter.h` says "call per block, not per sample" — but the
   current engine violates this inside `processBinauralForSource()`.

2. **SVFLowPass::setCoefficients() called per-sample** — head shadow and rear shadow SVFs also
   call `setCoefficients()` (containing `std::tan`) every sample inside the binaural loop.

3. **Multiple `std::pow(10, x/20)` and `std::sqrt` calls per sample** — chest bounce, floor
   bounce, near-field gain, and distance gain all compute dB-to-linear conversions per sample
   using `std::pow`.

4. **10 FeedbackCombFilters in series** — each requiring a ring-buffer push+read. At 192kHz/128
   samples this is 1280 comb reads per block just for depth processing.

5. **Stereo-width path doubles all binaural+distance work** — `processBinauralForSource()` runs
   twice per sample when `stereoActive` is true.

6. **Transcendental math in the per-sample stereo orbit** — `std::cos`, `std::sin`, `std::atan2`
   called per sample in the stereo orbit block for phase/offset smoothing.

**Primary recommendation:** Move coefficient recalculation out of the per-sample loop into a
per-block pre-computation step. This alone eliminates the most expensive transcendentals. Then
profile with MSVC's `/O2 /arch:AVX2` and target the remaining hot spots.

### Pattern 1: Per-Block Coefficient Pre-Computation (CRITICAL)

**What:** Compute all position-dependent filter targets once per block (outside the sample loop),
store in local variables, use inside the loop.

**When to use:** Any `setCoefficients()` call inside the per-sample `for (int i = 0; ...)` loop.

**Current violation in `processBinauralForSource()`:**
```cpp
// CURRENT (wrong — called once per sample from inside the loop):
presShelf.setCoefficients(dsp::BiquadType::HighShelf,
    currentParams.presenceShelfFreqHz, sr, 0.7071f, presenceGainDb);
earCanal.setCoefficients(dsp::BiquadType::PeakingEQ, ...);
pP1.setCoefficients(dsp::BiquadType::PeakingEQ, ...);
pN1.setCoefficients(dsp::BiquadType::PeakingEQ, ...);
pN2.setCoefficients(dsp::BiquadType::PeakingEQ, ...);
pShelf.setCoefficients(dsp::BiquadType::HighShelf, ...);
// PLUS: nfL.setCoefficients / nfR.setCoefficients per sample
// PLUS: shL.setCoefficients / shR.setCoefficients per sample
// PLUS: rSvfL.setCoefficients / rSvfR.setCoefficients per sample
```

**Target (correct — call once per block, read position-derived values that are block-constant):**
```cpp
// In the per-block preamble, OUTSIDE the sample loop:
presenceShelf_.setCoefficients(dsp::BiquadType::HighShelf,
    currentParams.presenceShelfFreqHz, sr, 0.7071f, presenceGainDb);
earCanalPeak_.setCoefficients(dsp::BiquadType::PeakingEQ, ...);
// etc.

// SVF head shadow — also per-block (azimuth changes per-block via smoothers,
// not per-sample unless LFO is on X-axis; even then smoothing makes
// per-sample SVF coeff calls redundant — the smoother handles the ramping):
shadowL_.setCoefficients(shadowCutoffTarget, sr);
shadowR_.setCoefficients(...);
```

**Caveat — LFO on X/Y/Z modulates position per sample:** The existing design recomputes
`modX/modY/modZ` per sample (for LFO modulation). Position-derived EQ values (presenceGainDb,
pinnaN1Freq, etc.) are therefore technically per-sample. However:
- With a 5ms smoothing time constant on the smoother and a 64-sample block at 44.1kHz (1.45ms),
  the change per sample is negligible.
- The correct optimization is: compute the per-sample modulated position (already done), but
  evaluate the biquad `setCoefficients()` only once at the start of the block using the
  **block-start modulated position**, not mid-block. The smoother already handles click-free
  transitions between blocks.
- This is safe because: biquads update once per block already (stated design goal from Phase 3
  decisions). The current violation is that `processBinauralForSource()` is called per-sample
  and calls `setCoefficients()` on every call.

**Fix strategy:** Refactor `processBinauralForSource()` to NOT call `setCoefficients()` internally.
Instead, compute all coefficient sets in the per-block preamble of `process()` and call
`setCoefficients()` there. The function then only calls `process()` on pre-configured filters.

### Pattern 2: Hoist Invariant Computations Out of Inner Loop

**What:** Identify values inside `for (int i = 0; i < numSamples; ++i)` that do not change per
sample and move them before the loop.

**Current per-sample computations that are loop-invariant:**
```cpp
// These are the same every iteration (params don't change mid-block):
const float modDist     = computeDistance(modX, modY, modZ);  // except when LFOs active
const float maxRange    = std::max(currentParams.sphereRadius - kMinDistance, 0.001f);
const float distRef     = currentParams.sphereRadius * kDistRefScale;
const float airCutoffMod = currentParams.airAbsMaxHz + ...;  // block-rate param change only
```

When LFOs are inactive (depth = 0), `modX/modY/modZ` are constant across the block, making all
of these truly loop-invariant. Even with LFOs active, chest/floor bounce targets and air LPF
coefficients change slowly enough that once-per-block is correct.

**Air LPF setCoefficients() called per-sample:**
```cpp
// In processDistanceForNode() — called per sample:
aL1.setCoefficients(airCutoffMod, sr);   // contains std::tan
aR1.setCoefficients(airCutoffMod, sr);   // same
aL2.setCoefficients(airCutoff2Mod, sr);
aR2.setCoefficients(airCutoff2Mod, sr);
```
These must move to per-block. Air absorption changes with distance, which changes with LFOs, but
the parameter is already smoothed over 150ms — per-block update is correct.

### Pattern 3: Eliminate Per-Sample dB-to-Linear Conversions

**What:** `std::pow(10.0f, x / 20.0f)` is expensive (~20ns on modern x86). These appear per-sample:

```cpp
// Per sample in the main loop:
const float chestLinearTarget = std::pow(10.0f, currentParams.chestGainDb / 20.0f) * chestElevNorm;
const float floorLinearTarget = std::pow(10.0f, currentParams.floorGainDb / 20.0f) * floorElevNorm;

// In processBinauralForSource() per sample:
const float nodeIldTarget = 1.0f - (1.0f - std::pow(10.0f, -currentParams.ildMaxDb / 20.0f)) * ...;
```

**Fix:** `currentParams.chestGainDb` and `currentParams.floorGainDb` don't change mid-block.
Pre-compute the linear gain in the per-block preamble:
```cpp
// Per-block preamble:
const float chestGainLin = std::pow(10.0f, currentParams.chestGainDb / 20.0f);
const float floorGainLin = std::pow(10.0f, currentParams.floorGainDb / 20.0f);
const float ildGainBase  = std::pow(10.0f, -currentParams.ildMaxDb / 20.0f);
// Then in loop: use chestGainLin * chestElevNorm (cheap multiply) instead of pow
```

### Pattern 4: Per-Sample Stereo Orbit Transcendentals

**What:** The stereo orbit block calls per sample (when stereoActive):
```cpp
phaseSmCos_ = std::cos(phaseRad) * b + phaseSmCos_ * angularSmA_;   // std::cos per sample
phaseSmSin_ = std::sin(phaseRad) * b + phaseSmSin_ * angularSmA_;   // std::sin per sample
offsetSmCos_ = std::cos(offsetRad) * b + ...;
offsetSmSin_ = std::sin(offsetRad) * b + ...;
const float smoothedPhase  = std::atan2(phaseSmSin_,  phaseSmCos_); // std::atan2 per sample
const float smoothedOffset = std::atan2(offsetSmSin_, offsetSmCos_);

// Also per orbit plane (when depth > 0):
const float cosXZ = std::cos(angXZ);   // std::cos per sample per active orbit
const float sinXZ = std::sin(angXZ);
```

**Fix:** `currentParams.stereoOrbitPhase` and `stereoOrbitOffset` are parameter values — they
don't change mid-block. Move the angular smoother advance to per-block. The angular smoother
coefficients are:
```
phaseSmCos_ = cos(phaseRad) * b + phaseSmCos_ * a
```
This is a first-order IIR on the cos/sin components. It's valid to update it once per block:
the 5ms smoothing time constant (default `kDefaultSmoothMs_Gain = 5ms`) means at 44.1kHz/128
samples = 2.9ms per block — one update per block is adequate. The `std::cos/sin` and `atan2`
calls then move to per-block, eliminating them from the inner loop.

### Pattern 5: Compiler Flag Optimization (High Value, Zero Risk)

**What:** The engine CMakeLists.txt currently has NO explicit optimization flags beyond what
`juce::juce_recommended_config_flags` provides. Adding Release-only flags:
- `/O2`: Full optimization (inlining, CSE, loop unrolling)
- `/arch:AVX2`: Enables 8-wide float SIMD auto-vectorization
- `/fp:fast`: Allows SIMD reordering, removes NaN/Inf strictness overhead

**Expected impact:** `/arch:AVX2` alone often provides 2-4x speedup on floating-point loops after
auto-vectorization kicks in for the filter chains. This requires no code changes.

```cmake
# engine/CMakeLists.txt
target_compile_options(xyzpan_engine PRIVATE
    $<$<CONFIG:Release>:/O2 /arch:AVX2 /fp:fast>
)
```

Note: `/fp:fast` is safe here because all NaN-producing paths are already guarded with
`std::isfinite()` checks in the existing code.

### Pattern 6: INFRA-03 — prepareToPlay Sample Rate Adaptation

**What:** INFRA-03 requires that changing the DAW sample rate triggers coefficient recalculation.
The current `prepare()` already handles this correctly: it recomputes all `setCoefficients()`
calls using `sr = static_cast<float>(inSampleRate)`. No coefficient is baked as a sample-rate
constant outside `prepare()`.

**Verification needed:** After the per-block optimization refactor, ensure that:
1. All filter coefficient sets use `sr` from the block context (not a captured value from
   `prepare()` time only).
2. `prepareToPlay()` calling `prepare()` re-initializes smoothers with correct `sr`.
3. A test processes a block at 44.1kHz, then calls `prepare()` with 96kHz, then processes
   again — output must not contain artifacts from stale coefficients.

### Anti-Patterns to Avoid

- **Over-SIMD-ing:** Writing manual AVX2 before confirming compiler auto-vectorization isn't
  sufficient. Profile first, then decide if manual intrinsics are needed.
- **Moving SVF setCoefficients to per-block for modulated filters:** SVFLowPass is designed for
  per-sample coefficient updates (TPT formulation is stable under modulation). Moving it to
  per-block is correct for head shadow (which is position-driven and block-rate), but if a future
  feature needs audio-rate SVF modulation, per-sample must be preserved.
- **Changing BiquadFilter to per-sample:** The existing design explicitly chose per-block biquad
  updates. This is a load-bearing decision — reverting it would require replacing biquads with
  SVFs in all EQ paths.
- **Allocating in process():** Zero allocations must be maintained (INFRA-04). No new `std::vector`
  or `new` in the hot path.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cycle-accurate CPU measurement | Custom RDTSC wrapper | `std::chrono::steady_clock` + block timing | Sufficient for block-level profiling; RDTSC requires careful serialization |
| SIMD biquad from scratch | Custom 4-wide or 8-wide biquad | Try `/arch:AVX2` auto-vectorization first | MSVC can auto-vectorize simple filter loops; manual intrinsics only if auto-vec fails |
| Approximate `std::tan` / `std::sin` | CORDIC or polynomial approx | The bottleneck is calling them per-sample at all; move to per-block, keep `std::` | The call site frequency, not the function cost, is the problem |

**Key insight:** The biggest wins come from algorithm restructuring (per-block vs per-sample),
not from replacing math functions with approximations. A `std::tan` called once per 64-sample
block is ~0.001% of CPU; a `std::tan` called 44100 times per second is ~2%.

## Common Pitfalls

### Pitfall 1: Refactoring processBinauralForSource() breaks stereo path

**What goes wrong:** `processBinauralForSource()` is called twice per sample in stereo mode — for
L and R channels with different node positions. If coefficient pre-computation is moved to the
per-block preamble, the code must compute two sets of coefficients (L-node and R-node based) and
apply them to the correct pipeline.

**Why it happens:** L node and R node have different X/Y/Z positions, meaning different
`presenceGainDb`, `pinnaN1Freq`, etc. If you compute one set of coefficients per block and use
them for both pipelines, the R pipeline gets wrong EQ values.

**How to avoid:** In the per-block preamble, compute L-node coefficients once (from `lNodeX/Y/Z`)
and R-node coefficients once (from `rNodeX/Y/Z`). Call `setCoefficients()` on the L pipeline's
filters with L-coefficients, and R pipeline's filters with R-coefficients, before entering the
sample loop. Use block-start positions for coefficient computation (the smoother handles
interpolation anyway).

**Warning signs:** Spatial image collapses, or L/R sound identical when stereo width is non-zero.

### Pitfall 2: Air LPF coefficient move breaks doppler path

**What goes wrong:** `airLPF_L/R.setCoefficients()` and the air cutoff computation currently
happen inside `processDistanceForNode()` which is called per-sample. Moving these to per-block
is correct for doppler-off mode. In doppler-on mode, `distFrac` changes per sample (because the
delay smoother advances per sample), so the air cutoff would ideally track it.

**How to avoid:** Accept the approximation: use block-start `distFrac` for air LPF coefficient
computation. The distance parameter is smoothed over 150ms — within a single 64-sample block
(1.45ms), the change in `distFrac` is negligible. Document this as an intentional approximation.

**Warning signs:** If distance changes very rapidly (test: large distance jump with doppler on),
verify air LPF transition sounds smooth, not stepped.

### Pitfall 3: `/fp:fast` causes NaN to propagate undetected

**What goes wrong:** `/fp:fast` can eliminate `isfinite()` checks or make NaN behavior
unpredictable. The existing code has two `isfinite` guards:
```cpp
if (!std::isfinite(dL)) dL = 0.0f;
if (!std::isfinite(dR)) dR = 0.0f;
```

**How to avoid:** These guards should remain. Under `/fp:fast`, `std::isfinite()` still works
correctly on MSVC — it compiles to a bitfield check, not a floating-point comparison. Verify in
Release build with a NaN-injection test that the guards still fire.

### Pitfall 4: INFRA-03 regression after coefficient refactor

**What goes wrong:** After moving coefficient computation to per-block, a stale `sr` value is
captured (e.g., from prepare() time) rather than recomputed each block from `sampleRate_`.

**How to avoid:** Always derive `const float sr = static_cast<float>(sampleRate);` at the top of
`process()` (already done), and use this `sr` in all per-block `setCoefficients()` calls. When
`prepare()` is called with a new sample rate, the next `process()` call will use the new `sr`
automatically.

### Pitfall 5: Per-block coefficient update artifacts during automation

**What goes wrong:** Moving EQ coefficient updates from per-sample to per-block introduces a
step change at block boundaries — the filter coefficients jump at each block boundary instead of
interpolating smoothly.

**Why it matters:** For slow parameter changes (automation), this is inaudible (the step is
sub-sample in perceptual terms). For rapid parameter changes (LFO depth at high rate), it can
cause zipper-like artifacts.

**How to avoid:** The existing OnePoleSmooth infrastructure already handles this for gain and
delay parameters. For filter cutoff frequency parameters that feed into EQ biquads, the
position-derived values (presenceGainDb, pinnaN1Freq) are already smoothed implicitly by the
fact that position changes come through parameter smoothers. The LFO modulation on X/Y/Z is the
concern — but at typical LFO rates (0.01–50Hz) and 64-sample blocks (1.45ms), block-rate biquad
updates are indistinguishable from per-sample updates.

## Code Examples

Verified patterns from the existing codebase:

### Per-Block Coefficient Pre-Computation (Target Pattern)

```cpp
// In XYZPanEngine::process() BEFORE the per-sample loop:
// (sr is already computed at top of process())

// Compute block-start modulated position for EQ targets
// (use currentParams.x/y/z directly; LFO delta will be applied inside loop
//  but the coefficient-driving EQ targets use block-start position)
const float blkDist = computeDistance(currentParams.x, currentParams.y, currentParams.z);
const float blkHorizMag = std::sqrt(currentParams.x * currentParams.x + currentParams.y * currentParams.y);
const float blkRearFactor = (blkHorizMag > 1e-7f)
    ? (-currentParams.y / blkHorizMag) : currentParams.y;
const float blkPresenceGainDb = currentParams.presenceShelfMaxDb * (-blkRearFactor);
const float blkEarCanalGainDb = std::min(0.0f, currentParams.earCanalMaxDb * (-blkRearFactor));
const float blkPinnaGainDb    = -15.0f + 20.0f * std::max(0.0f, currentParams.z);
const float blkShelfGainDb    = 3.0f * std::clamp(currentParams.z + 1.0f, 0.0f, 1.0f);
const float blkPinnaN1Freq    = kPinnaN1MinHz + (kPinnaN1MaxHz - kPinnaN1MinHz)
    * ((currentParams.z + 1.0f) * 0.5f);

// Set biquad coefficients ONCE per block:
presenceShelf_.setCoefficients(dsp::BiquadType::HighShelf,
    currentParams.presenceShelfFreqHz, sr, 0.7071f, blkPresenceGainDb);
earCanalPeak_.setCoefficients(dsp::BiquadType::PeakingEQ,
    currentParams.earCanalFreqHz, sr, currentParams.earCanalQ, blkEarCanalGainDb);
pinnaP1_.setCoefficients(dsp::BiquadType::PeakingEQ,
    kPinnaP1FreqHz, sr, kPinnaP1Q, kPinnaP1GainDb);
pinnaNotch_.setCoefficients(dsp::BiquadType::PeakingEQ,
    blkPinnaN1Freq, sr, currentParams.pinnaNotchQ, blkPinnaGainDb);
pinnaNotch2_.setCoefficients(dsp::BiquadType::PeakingEQ,
    blkPinnaN1Freq + kPinnaN2OffsetHz, sr, kPinnaN2Q, kPinnaN2GainDb);
pinnaShelf_.setCoefficients(dsp::BiquadType::HighShelf,
    currentParams.pinnaShelfFreqHz, sr, 0.7071f, blkShelfGainDb);

// Pre-compute dB-to-linear conversions (expensive pow):
const float chestGainLin = std::pow(10.0f, currentParams.chestGainDb / 20.0f);
const float floorGainLin = std::pow(10.0f, currentParams.floorGainDb / 20.0f);
const float ildGainBase  = std::pow(10.0f, -currentParams.ildMaxDb / 20.0f);

// Air LPF targets from block-start distance:
const float blkMaxRange   = std::max(currentParams.sphereRadius - kMinDistance, 0.001f);
const float blkDistFrac   = std::clamp((blkDist - kMinDistance) / blkMaxRange, 0.0f, 1.0f);
const float blkAirCutoff1 = currentParams.airAbsMaxHz
    + (currentParams.airAbsMinHz - currentParams.airAbsMaxHz) * blkDistFrac;
const float blkAirCutoff2 = kAirAbs2MaxHz
    + (kAirAbs2MinHz - kAirAbs2MaxHz) * blkDistFrac;
airLPF_L_.setCoefficients(blkAirCutoff1, sr);
airLPF_R_.setCoefficients(blkAirCutoff1, sr);
airLPF2_L_.setCoefficients(blkAirCutoff2, sr);
airLPF2_R_.setCoefficients(blkAirCutoff2, sr);
```

### Block-Level Performance Microbenchmark (TestPerformance.cpp)

```cpp
#include <catch2/catch_test_macros.hpp>
#include "xyzpan/Engine.h"
#include <chrono>
#include <vector>

TEST_CASE("Engine throughput — 128 samples at 44.1kHz") {
    xyzpan::XYZPanEngine engine;
    engine.prepare(44100.0, 128);

    std::vector<float> inBuf(128, 0.5f);
    std::vector<float> outL(128), outR(128);
    std::vector<float> auxL(128), auxR(128);
    const float* inputs[1] = { inBuf.data() };

    xyzpan::EngineParams params;
    params.x = 0.3f; params.y = 0.8f; params.z = 0.1f;
    engine.setParams(params);

    // Warm up
    for (int i = 0; i < 100; ++i)
        engine.process(inputs, 1, outL.data(), outR.data(), nullptr, nullptr, 128);

    constexpr int kTrials = 1000;
    auto t0 = std::chrono::steady_clock::now();
    for (int i = 0; i < kTrials; ++i)
        engine.process(inputs, 1, outL.data(), outR.data(), nullptr, nullptr, 128);
    auto t1 = std::chrono::steady_clock::now();

    const double usPerBlock = std::chrono::duration<double, std::micro>(t1 - t0).count() / kTrials;
    // Budget: 128/44100 = 2902 microseconds. Target: < 10% CPU = < 290us per block.
    // At 44.1kHz, 128 samples = 2.9ms. Budget at 10% CPU = 0.29ms = 290us.
    INFO("us per block: " << usPerBlock);
    CHECK(usPerBlock < 290.0);  // 10% CPU budget at 128/44.1kHz
}
```

## CPU Budget Analysis

**Target:** Plugin runs at < 10% CPU on mid-range hardware at 128-sample buffers.

**Mid-range hardware baseline:** Ryzen 5 3600 (~4GHz, AVX2) or Intel Core i5-9400.

**Block budget at 44.1kHz, 128 samples:**
- Block duration = 128 / 44100 = 2.903ms
- 10% CPU budget = 0.290ms = 290 microseconds per block

**Per-sample cost estimate (BEFORE optimization, Release build):**

| Operation | Count/sample (mono mode) | Estimated cost (ns) | Total/sample (ns) |
|-----------|--------------------------|--------------------|--------------------|
| BiquadFilter setCoefficients (cos+sin+pow+sqrt) | 6 | ~50ns ea | ~300ns |
| SVFLowPass setCoefficients (tan) | 4 | ~20ns ea | ~80ns |
| Air LPF setCoefficients (tan) | 4 | ~20ns ea | ~80ns |

At 44100 Hz: 700ns * 44100 = 30.87ms per second = ~3% CPU (mono mode)
At 128 samples: 700ns * 128 = 89.6us per block

**The ~300ns from biquad setCoefficients alone would drop to near-zero after per-block fix.**

In stereo mode (stereoActive=true), all binaural + distance work doubles: ~6% CPU.

With `/arch:AVX2` auto-vectorization on the filter chains, expect an additional 2-4x improvement
on the pure-arithmetic paths, bringing stereo mode below the 10% target comfortably.

## Optimization Sequence (Priority Order)

The following is the recommended sequence, highest value first:

**1. Per-block coefficient pre-computation** — eliminates ~300ns/sample of transcendentals.
   Zero risk to correctness if done carefully. The INFRA-03 requirement is satisfied because
   `sr` is always derived from the current-block `sampleRate` member.

**2. Compiler flags: `/O2 /arch:AVX2 /fp:fast`** — no code changes required. Expected 2-4x on
   auto-vectorizable loops. Apply to `xyzpan_engine` target in Release configuration.

**3. Per-sample orbit transcendentals to per-block** — eliminates cos/sin/atan2 in the stereo
   orbit computation. Medium complexity change to the angular smoother logic.

**4. Per-block air LPF coefficients** — simple extraction from `processDistanceForNode()`.

**5. Per-block dB-to-linear pre-computation** — trivial; move 3x `std::pow` before the loop.

**6. Profile with VS Diagnostics in Release** — verify the above accounted for the bottleneck.
   Look for any remaining hot loops for manual SIMD if auto-vectorization missed them.

## Open Questions

1. **Does JUCE's `juce_recommended_config_flags` already include `/O2`?**
   - What we know: JUCE's CMake module adds recommended flags; exact flags not audited.
   - What's unclear: Whether `/arch:AVX2` and `/fp:fast` are included.
   - Recommendation: Check `cmake --build build --config Release -- /p:CompileFlags=1` output or
     inspect JUCE's `juce_recommended_config_flags` target. Add explicit flags only if missing.

2. **Is the comb bank the next bottleneck after biquad setCoefficients is fixed?**
   - What we know: 10 series comb filters with ring-buffer push+read per sample.
   - What's unclear: Whether MSVC auto-vectorizes the comb loop (depends on aliasing analysis).
   - Recommendation: After removing per-sample setCoefficients calls, profile again. If combs
     are the next hot spot, consider a batch-processing version that processes all 10 combs
     across a block at once (enables vectorization across samples).

3. **Does stereo-width mode create a doubling problem that makes the 10% budget impossible?**
   - What we know: Stereo mode runs binaural+distance pipeline twice.
   - What's unclear: Real-world CPU at 128 samples with stereo active + all LFOs.
   - Recommendation: Measure actual baseline before optimizing. If stereo mode is 2x the budget,
     the per-block coefficient fix alone likely resolves it (the setCoefficients calls dominate).
